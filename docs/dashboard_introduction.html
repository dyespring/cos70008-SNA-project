Dashboard Introduction

Text-to-Network Engine — Dashboard at a glance

A one-page introduction to the Streamlit dashboard that sits on top of the Neo4j conceptual network: what each of the eight tabs is for, who it is aimed at, and the quickest way to read it.

1. What this dashboard is

The dashboard is the analyst-facing front end of the Text-to-Network Engine. The pipeline (pipeline.py) extracts concepts and relationships from unstructured text (policy documents, Yelp reviews, or a combined corpus), writes them into Neo4j, and runs Graph Data Science (GDS) algorithms server-side. The dashboard reads everything back through Cypher and presents it as eight focused tabs.

How to launch

From the project root, after Neo4j is up and the pipeline has been run:

  • python -m streamlit run src/extensions/dashboard.py
  • Pick a source in the sidebar — policy, yelp, or combined.
  • Optionally set an LLM_PROVIDER in .env to enable the "Polish with LLM" buttons; otherwise everything still works in deterministic mode.

Tabs in this dashboard

  1. Summary — executive snapshot
  2. Network — the live graph itself
  3. Insights — Key / Risk / Action cards
  4. Centrality — who is influential, and how
  5. Communities — themes inside the network
  6. Brokers — concepts that bridge themes
  7. Temporal — how the story changes over time
  8. Chat — conversational Q&A grounded in the graph

2. Each tab, in one card

1 Summary

For: anyone — a one-screen snapshot for stakeholders.

What it shows. Headline counts of concepts and relationships, network density, average clustering, and the number of weakly-connected components. On combined data it also breaks down concept counts by corpus (policy / Yelp / shared).

Purpose. Give a 10-second answer to “how big is the network and how connected is it?”, then point readers at the Insights tab for the full card deck.

2 Network

For: visual analysts — explore the live graph in the browser.

What it shows. An interactive vis-network canvas rendered straight from Neo4j (no NetworkX rebuild). Controls let you change Top-N, ranking metric (PageRank, betweenness, degree…), edge weight cut-off, colour-by (community or source), the temporal slice, and the physics solver.

Purpose. A scoped, high-impact subgraph you can pan, zoom and inspect — without dumping the entire database into the browser.

3 Insights

For: decision-makers — narrative findings, not raw tables.

What it shows. Three columns of insight cards driven by Cypher over the GDS metrics already on each node:

  • Key — descriptive (hubs, bridges, isolated clusters, cross-source overlap).
  • Risk — diagnostic (single points of failure, echo chambers, negative-sentiment clusters, fragmentation).
  • Action — prescriptive (diversify, leverage bridges, investigate isolates).

An optional Polish with LLM toggle rewrites each card in executive-friendly language.

Purpose. Turn graph metrics into actionable observations a non-technical audience can read.

4 Centrality

For: analysts asking “who matters?”.

What it shows. Leaderboards under five lenses (PageRank, betweenness, degree, closeness, eigenvector) plus a two-axis scatter to spot concepts that score high on more than one. An optional LLM blurb turns the tables into a short brief.

Purpose. Identify which concepts are popular, load-bearing, or both — the kind of vocabulary worth a closer look in a meeting.

5 Communities

For: analysts mapping themes.

What it shows. Louvain communities with two anchor lists per cluster (top by frequency vs. top by PageRank), a size distribution, optional cohesion (how self-contained a theme is), and a drill-down: members, sample internal edges, and which other communities a cluster reaches.

Purpose. See the conversation themes hidden in the text and how porous or self-contained each one is.

6 Brokers

For: anyone hunting bridge vocabulary.

What it shows. Concepts ranked by betweenness × cross_community_edges — the “bilingual” ideas that link one Louvain community to another. A drill-down shows a broker’s home community, which clusters it reaches, and its strongest cross-community edges. On combined data there is also a second list of concepts whose neighbours span policy and Yelp corpora.

Purpose. Surface the vocabulary that connects otherwise separate storylines — often the most interesting concepts in a network.

7 Temporal

For: anyone tracking change over time.

What it shows. Three angles on slice-tagged subgraphs in Neo4j (built by pipeline.py temporal --source <src> --temporal N):

  • Trend — is the network growing, getting denser, shifting sentiment?
  • Drift — between neighbouring slices, what vocabulary appears, disappears, and how much overlap remains?
  • Comparison — first vs. last slice, the overall shift.

A Trend explorer below lets you filter slices and metrics for presentations; insight cards can optionally be polished with the LLM.

Purpose. Show how the conceptual landscape evolves — useful for policy timelines or year-on-year Yelp reviews.

8 Chat

For: anyone who wants to ask the graph a question.

What it shows. A chatbot grounded in the Neo4j graph. The router pulls Cypher snippets (top concepts, communities, paths, descriptions) and optional vector hits over node / edge embeddings, then passes them to the configured LLM provider (echo, openai, dashscope, huggingface_api, huggingface_local).

Purpose. Natural-language exploration of the graph — “Who recommended the food?”, “What bridges communities 2 and 5?”, “Top concepts by betweenness.” The echo provider is a zero-dependency fallback that returns the retrieved graph context verbatim.

3. Where the numbers come from

TabBacked by
Summary Cypher counts + pipeline.py analyse GDS metrics on :Concept.
Network cypher_to_vis_json → vis_network_html rendered with vis-network.
Insights src/extensions/insight_engine.py + optional LLM polish via metric_llm.
Centrality gds.pageRank, gds.betweenness, gds.closeness, gds.eigenvector, gds.degree written back as node properties.
Communities gds.louvain + cohesion / drill-down Cypher in neo4j_graph_context.
Brokers Cypher: high-betweenness × cross-community edge counts; cross-source bridges on combined.
Temporal pipeline.py temporal writes slice_id on nodes/edges; temporal_insights.py reads them back.
Chat chatbot.QueryRouter + Neo4jGraphContext + optional node/edge vector indexes.
