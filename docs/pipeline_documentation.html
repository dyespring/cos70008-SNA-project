1. Architecture Overview

3. Pipeline stages (1–6) + commands (7–8)

Stages 1–6 are the core batch path. Stages 7–8 are separate CLI subcommands that extend or query data already in Neo4j.

1 Ingestion

_load_documents() in pipeline.py

  • Policy: pdf_reader.read_policy_pdf() extracts text per page using pdfplumber, strips headers/footers
  • Yelp: yelp_reader.load_yelp_reviews() reads extracted JSONL, filters by category, samples N reviews
  • Combined: both are loaded into the same document list

2 Preprocessing

Same path as stage 1: _load_documents() applies cleaning then spaCy tokenisation

  • clean_text() — removes URLs, emails, Unicode artefacts, policy-specific noise
  • SpacyTokeniser — tokenises, lemmatises, tags POS, extracts NER entities & noun chunks
  • Each document gets a source tag ("policy" or "yelp")

3 Extraction

cmd_etl() · src/extraction/

  • ConceptExtractor — 3 strategies: Named Entity Recognition, Noun-Phrase chunking, TF-IDF keywords
  • Each concept tracks source_origins → knows if it came from policy, yelp, or both
  • RelationshipExtractor — co-occurrence windows + dependency parsing; classifies as ASSOCIATION / ACTION / CAUSATION
  • Optional: advanced.refine_concepts() when --advanced-extraction is set (src/extraction/advanced.py)

4 Write to Neo4j

pipeline.py etl · src/network/neo4j_writer.py

  • Neo4jGraphWriter upserts Concept[] and Relationship[] via batched Cypher UNWIND
  • Edges below --min-weight are pruned before the write
  • Parallel edges of different types are aggregated; verb counters preserved as r.top_verb / r.verb_count / r.verb_list
  • Scoped per source_label (and optional slice_id) so multiple datasets coexist in one database
  • Optional CSV dumps via --export-csv on the viz / temporal subcommands

5 GDS Analysis (server-side)

pipeline.py analyse · src/network/gds_analyser.py

  • Centrality: gds.degree.write, gds.pageRank.write, gds.betweenness.write, gds.closeness.write, gds.eigenvector.write
  • Communities: gds.louvain.write (writes c.community); gds.wcc.write for weakly-connected components
  • Brokerage: Cypher-only — high-betweenness nodes whose neighbours span multiple communities
  • Cross-source (combined mode): overlap %, bridging concepts, cross-source edges via Cypher aggregations
  • Each algorithm projects an in-memory graph scoped per source_label, runs, writes back, then drops the projection

6 Visualisation

pipeline.py viz · src/visualisation/cypher_to_nx.py

  • fetch_subgraph() pulls the top-N nodes by PageRank (and the edges between them) back into a transient nx.DiGraph for rendering
  • Static: network graph, top concepts bar chart, community distribution, co-occurrence heatmap, source-coloured network, source overlap chart
  • Interactive: pyvis HTML network with hover tooltips, zoom, drag (written to results/ by viz)
  • Combined mode colours nodes by source: Policy Yelp Both
  • The Streamlit dashboard’s in-app graph uses vis-network (vis_network_html.py), not this pyvis file — same Neo4j data, different renderer.

3b. Commands — stage 7 (temporal) & stage 8 (chat)

7 Temporal comparison

pipeline.py temporal · src/extensions/temporal.py (+ insights in temporal_insights.py)

  • Splits the corpus into N time-ordered slices; writes subgraphs scoped by slice_id on nodes and edges
  • Runs GDS per slice unless --skip-gds (faster iteration; some dashboard temporal cards fall back to frequency-only)
  • Pairwise overlap / Jaccard style metrics via Cypher; optional CSV via --export-csv

8 Chatbot (LLM / RAG)

pipeline.py chat · chat.py · src/extensions/chatbot.py

  • Neo4jGraphContext supplies Cypher-backed snippets; optional Neo4jVectorStore + Neo4jEdgeVectorStore when --vector-store
  • Pluggable LLM providers (LLM_PROVIDER env); --provider overrides for a single session
  • Delegates the interactive loop to chat.py — same grounding model as the dashboard Chat tab

5. Combined Mode & Cross-Source Analysis

When --source combined is used, the pipeline ingests both data sources into a unified corpus. Every concept is tagged with its origin, enabling cross-domain analysis.

Cross-Source Metrics

  • source_distribution — count of concepts per source type
  • source_comparison_summary — policy-only vs yelp-only vs shared, overlap percentage
  • cross_source_edges — every edge connecting nodes from different sources
  • bridging_concepts — top concepts ranked by bridge_score = betweenness × (1 + cross-source neighbour ratio)

6. CLI Subcommands & Arguments

Every invocation takes the form python pipeline.py <subcommand> [options]. Neo4j must be reachable; the pipeline does not run without it.

8. Usage Examples

# Bring up Neo4j (one-time per machine)
docker compose up -d

# Combined pipeline — 500 Yelp reviews + policy, end-to-end
python pipeline.py all --source combined --reset --embed \
    --yelp-sample 500 --min-weight 2 --output results/combined_500

# Larger batch — 5000 reviews, stronger edge filter
python pipeline.py all --source combined --reset --embed \
    --yelp-sample 5000 --min-weight 3 --output results/combined_5000

# Policy only, with sentiment-weighted edges
python pipeline.py all --source policy --reset --sentiment \
    --output results/policy_only

# Yelp only, specific category
python pipeline.py all --source yelp --reset \
    --yelp-sample 1000 --yelp-category "Food" --output results/yelp_food

# Use only NER + TF-IDF (skip noun phrases)
python pipeline.py etl --source combined --reset --methods ner tfidf
python pipeline.py analyse --source combined
python pipeline.py viz --source combined --top-n 200 --export-csv

# Temporal slicing
python pipeline.py temporal --source combined --temporal 3 --export-csv

# Chat
python pipeline.py chat --source combined --vector-store
